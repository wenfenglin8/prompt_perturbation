How PDR is calculated in the current RQ1 script

The current report uses a loss-based PDR calculation.

For each case:

clean_mean_performance = mean correctness across repeated original-prompt generations
perturbed_mean_performance = mean correctness across repeated perturbed-prompt generations

clean_loss = 1 - clean_mean_performance
perturbed_loss = 1 - perturbed_mean_performance

uncorrected_pdr_loss = clean_loss
corrected_pdr = perturbed_loss - clean_loss

Equivalently:

corrected_pdr = (1 - perturbed_mean_performance) - (1 - clean_mean_performance)
              = clean_mean_performance - perturbed_mean_performance

Important interpretation:

- uncorrected_pdr_loss is the loss under the original clean prompt.
- perturbed_pdr_loss is the loss under the perturbed prompt.
- corrected_pdr is the additional loss caused by the perturbation relative to the original prompt.
- If the original prompt is compared against itself, corrected_pdr is 0 by definition.

Code locations:

- promptrobust_reference_pdr_eval.py:427-428 defines performance_loss(performance) = 1.0 - performance.
- promptrobust_reference_pdr_eval.py:622-626 computes clean_mean, perturbed_mean, clean_loss, perturbed_loss, and corrected_pdr.
- promptrobust_reference_pdr_eval.py:673-677 computes dataset-level clean repeated performance, perturbed repeated performance, uncorrected PDR loss, perturbed loss, and corrected PDR.

Plain Chinese explanation of clean_mean_performance and perturbed_mean_performance

clean_mean_performance 的意思：

对同一个原始 prompt，不只让模型回答一次，而是重复生成多次。
每一次回答都会被判定为一个正确性分数。

例如 samples = 5 时，同一个原始 prompt 会得到 5 个回答：

回答1: 正确 = 1
回答2: 正确 = 1
回答3: 错误 = 0
回答4: 正确 = 1
回答5: 错误 = 0

那么：

clean_mean_performance = (1 + 1 + 0 + 1 + 0) / 5 = 0.6

也就是说，在原始 prompt 下，这个 case 的平均正确率是 0.6。

perturbed_mean_performance 的意思：

把同一个任务的 prompt 做扰动，例如 paraphrase、reordering、formatting、context injection 或 surface noise。
然后也让模型对这个扰动后的 prompt 重复回答多次。
每一次扰动 prompt 的回答也会被判定为正确或错误。

例如同一个 case 的扰动 prompt 也生成 5 次：

回答1: 正确 = 1
回答2: 错误 = 0
回答3: 错误 = 0
回答4: 正确 = 1
回答5: 错误 = 0

那么：

perturbed_mean_performance = (1 + 0 + 0 + 1 + 0) / 5 = 0.4

也就是说，在扰动 prompt 下，这个 case 的平均正确率是 0.4。

然后 PDR 比较的就是：

原始 prompt 的平均正确率 clean_mean_performance
和
扰动 prompt 的平均正确率 perturbed_mean_performance

如果原始 prompt 平均正确率是 0.6，扰动 prompt 平均正确率是 0.4：

clean_loss = 1 - 0.6 = 0.4
perturbed_loss = 1 - 0.4 = 0.6
corrected_pdr = 0.6 - 0.4 = 0.2

这个 0.2 表示：扰动 prompt 让性能额外下降了 20 个百分点。

关键点：

clean_mean_performance 不是单次原始 prompt 的结果，而是原始 prompt 重复生成后的平均正确率。
perturbed_mean_performance 也不是单次扰动 prompt 的结果，而是扰动 prompt 重复生成后的平均正确率。
这样做是为了避免只看一次回答时，被模型本身的随机波动误导。

Why average clean single-sample performance is only 0.6213

Question:

Average clean single-sample performance = 0.6213. Why is it so low? Are all tasks this low?

Answer:

No. The low overall average does not mean every task has only about 62% accuracy.
After splitting by task, factual QA is very high, while math reasoning and code generation are much lower.

Task-level clean performance from results/pdr_metrics_rq1_pdr_newloss_three_task_50x5_combined.csv:

task              n     clean_single    perturbed_single    clean_repeated    perturbed_repeated
factual_qa        250   0.9800          0.9787              0.9789            0.9770
math_reasoning    250   0.3680          0.3880              0.3944            0.3984
code_generation   250   0.5160          0.4840              0.5104            0.4784

Therefore:

The overall clean single-sample performance is:

(0.9800 + 0.3680 + 0.5160) / 3 = 0.6213

So the 0.6213 aggregate is low because it averages three very different tasks:

- factual_qa is easy/high-performing in this setup: clean single = 0.9800
- math_reasoning is hard/low-performing: clean single = 0.3680
- code_generation is moderate but still low: clean single = 0.5160

Chinese explanation:

0.6213 不是说每个任务都只有 62% 正确率。
它是 factual QA、math reasoning、code generation 三类任务混合后的平均值。

其中：

factual QA 的原始 prompt 单次正确率是 0.9800，非常高。
math reasoning 的原始 prompt 单次正确率是 0.3680，很低。
code generation 的原始 prompt 单次正确率是 0.5160，也不高。

所以总平均被 math_reasoning 和 code_generation 拉低了。

为什么 math 和 code 低：

1. math_reasoning 使用的是 MATH 风格题目，本身难度较高。模型需要完整推理和最终答案匹配，错误空间很大。
2. code_generation 使用 HumanEval 风格任务，判分通常是通过测试用例或严格函数行为判断。代码只要格式、边界条件、函数实现有问题，就会被判错。
3. factual_qa 使用 SQuAD V2 风格问答，当前样本下模型表现很稳定，所以正确率接近 0.98。

Important caution:

The aggregate 0.6213 should not be interpreted alone as "the model is only 62% accurate on every task."
A better paper/report description is:

The aggregate clean single-sample performance is 0.6213, but this average masks strong task heterogeneity: factual QA is near ceiling at 0.9800, whereas math reasoning and code generation are substantially lower at 0.3680 and 0.5160, respectively.

The official stored code correctness has a major false-negative issue.
Many model outputs are valid function bodies inside Markdown code fences, for example:

The current evaluator strips the code fence using a regex that also removes the leading indentation on the first code line. Then it appends the result to the HumanEval prompt prefix. This can produce:

- IndentationError
- SyntaxError: 'return' outside function

even when the function body is logically correct.

Recommendation: using PDR/correctness to estimate model sample noise

User question:

Can we get model sample noise from PDR, analogous to how similarity-based sample noise is computed?

Short answer:

Yes. For PDR/correctness, the cleanest approach is to define sample noise as within-prompt variability in correctness across repeated generations.

Similarity sample noise uses output distances among repeated generations from the same prompt.
PDR sample noise should analogously use correctness/performance differences among repeated generations from the same prompt.

Recommended definition:

For each case and prompt version, we have repeated correctness scores:

clean prompt:

c_1, c_2, c_3, c_4, c_5

where each c_i is usually 0 or 1 for math/code, and can be continuous for factual QA.

Define correctness sample noise as the average pairwise absolute difference:

PDR_sample_noise_case = average over i<j of |c_i - c_j|

For binary correctness, this has an intuitive meaning:

- If all 5 outputs are correct: [1,1,1,1,1], sample noise = 0
- If all 5 outputs are wrong: [0,0,0,0,0], sample noise = 0
- If outputs are mixed, e.g. [1,1,1,0,0], sample noise > 0

For n samples with k correct and n-k wrong, binary pairwise sample noise is:

PDR_sample_noise = k * (n-k) / C(n,2)

For n=5:

[1,1,1,1,1] -> 0/10 = 0.0
[1,1,1,1,0] -> 4/10 = 0.4
[1,1,1,0,0] -> 6/10 = 0.6
[1,1,0,0,0] -> 6/10 = 0.6
[1,0,0,0,0] -> 4/10 = 0.4
[0,0,0,0,0] -> 0/10 = 0.0

This is better than standard deviation for reporting because it directly matches the similarity-sample-noise idea: pairwise within-prompt variability.

How to connect it to PDR:

PDR is based on performance loss:

clean_loss = 1 - mean(clean correctness)
perturbed_loss = 1 - mean(perturbed correctness)
corrected_PDR = perturbed_loss - clean_loss

The corresponding sample-noise baseline should not be subtracted from loss directly as if it were loss.
Instead, it should be reported as uncertainty/instability in the correctness estimate.

Recommended paired quantities:

1. Clean repeated performance:

mean(c_1,...,c_5)

2. Clean repeated loss / uncorrected PDR:

1 - mean(c_1,...,c_5)

3. Clean correctness sample noise:

average pairwise |c_i - c_j|

4. Perturbed repeated performance:

mean(p_1,...,p_5)

5. Corrected PDR:

mean(clean correctness) - mean(perturbed correctness)

6. Perturbed correctness sample noise:

average pairwise |p_i - p_j|

7. Optional pooled sample noise:

PDR_sample_noise_pooled = average(clean_sample_noise, perturbed_sample_noise)

For comparing perturbation effect to sample noise:

Use a ratio or contrast such as:

effect_to_noise_ratio = abs(corrected_PDR) / pooled_PDR_sample_noise

or:

noise_adjusted_PDR_signal = abs(corrected_PDR) - pooled_PDR_sample_noise

But be careful:

If using subtraction, call it "signal beyond within-prompt correctness variability", not "corrected PDR loss".

Recommended statistical test:

Use bootstrap resampling over the 5 clean and 5 perturbed samples for each case:

1. Resample 5 clean outputs with replacement.
2. Resample 5 perturbed outputs with replacement.
3. Compute clean_mean, perturbed_mean, corrected_PDR.
4. Repeat many times.
5. Report confidence interval for corrected_PDR.
6. Estimate probability that corrected_PDR > 0.

This is stronger than only reporting a sample-noise scalar.

Recommended implementation plan:

Add these columns to the recomputed metrics CSV:

- clean_correctness_pairwise_noise
- perturbed_correctness_pairwise_noise
- pooled_correctness_pairwise_noise
- abs_corrected_pdr
- effect_to_noise_ratio
- pdr_signal_beyond_noise = abs_corrected_pdr - pooled_correctness_pairwise_noise
- pdr_signal_beyond_noise_clipped = max(0, abs_corrected_pdr - pooled_correctness_pairwise_noise)

For task-level reporting:

Aggregate by task:

- mean clean repeated performance
- mean perturbed repeated performance
- mean corrected PDR
- mean clean correctness sample noise
- mean perturbed correctness sample noise
- mean pooled correctness sample noise
- mean effect-to-noise ratio

Important interpretation:

Do not say:

"sample noise is additional PDR loss"

Better wording:

"Correctness sample noise measures within-prompt variability in task success across repeated generations. It quantifies how unstable the performance estimate is even before perturbation. Corrected PDR measures the mean clean-vs-perturbed performance difference. Comparing corrected PDR with correctness sample noise indicates whether the perturbation effect is large relative to ordinary repeated-generation variability."

Best recommendation for this project:

Use pairwise correctness sample noise as the PDR analogue of embedding-distance sample noise.

Formula:

correctness_sample_noise = average_{i<j} |score_i - score_j|

Then report corrected PDR and sample noise side by side, and optionally add bootstrap confidence intervals for corrected PDR.

Why [1,1,1,1,0] has sample noise = 0.4

User question:

Why is sample noise 0.4 for [1,1,1,1,0]?

Answer:

The recommended correctness sample noise is the average pairwise absolute difference:

sample_noise = average_{i<j} |score_i - score_j|

For 5 repeated outputs, there are:

C(5,2) = 5 * 4 / 2 = 10

pairwise comparisons.

For:

[1, 1, 1, 1, 0]

label the five outputs as:

s1 = 1
s2 = 1
s3 = 1
s4 = 1
s5 = 0

All 10 pairwise differences are:

|s1 - s2| = |1 - 1| = 0
|s1 - s3| = |1 - 1| = 0
|s1 - s4| = |1 - 1| = 0
|s1 - s5| = |1 - 0| = 1

|s2 - s3| = |1 - 1| = 0
|s2 - s4| = |1 - 1| = 0
|s2 - s5| = |1 - 0| = 1

|s3 - s4| = |1 - 1| = 0
|s3 - s5| = |1 - 0| = 1

|s4 - s5| = |1 - 0| = 1

Sum of pairwise differences:

0 + 0 + 0 + 1 + 0 + 0 + 1 + 0 + 1 + 1 = 4

Average pairwise difference:

4 / 10 = 0.4

So:

[1,1,1,1,0] -> sample noise = 0.4

Intuition:

There are 4 correct outputs and 1 wrong output.
The wrong output disagrees with each of the 4 correct outputs.
So there are 4 disagreeing pairs out of 10 total pairs.

Therefore:

sample_noise = 4 / 10 = 0.4

General binary formula:

If there are n repeated outputs and k are correct, then:

number of correct-wrong disagreeing pairs = k * (n-k)
number of total pairs = C(n,2)

So:

sample_noise = k * (n-k) / C(n,2)

For [1,1,1,1,0]:

n = 5
k = 4

sample_noise = 4 * (5-4) / C(5,2)
             = 4 * 1 / 10
             = 0.4

1. clean baseline loss:

这是原始 clean prompt 下模型本来就会犯错的部分。
它可以理解为模型本身能力、任务难度、采样随机性、评价器误差等共同造成的 baseline loss。

公式：

clean_performance = mean(clean repeated correctness)
clean_baseline_loss = 1 - clean_performance

2. perturbation-induced loss:

这是扰动 prompt 相比 clean prompt 额外造成的性能下降。
这才是更接近“由 perturbation 带来的 loss”的部分。

公式：

perturbed_performance = mean(perturbed repeated correctness)
perturbed_loss = 1 - perturbed_performance
perturbation_induced_loss = perturbed_loss - clean_baseline_loss

等价地：

perturbation_induced_loss = clean_performance - perturbed_performance

也就是当前代码里的 corrected_PDR。

3. sampling instability / sample noise:

这是同一个 prompt 多次生成时，正确性本身是否稳定。
它不应该被解释成额外 loss，而应该解释成 performance estimate 的不稳定性。

公式：

correctness_sample_noise = average_{i<j} |score_i - score_j|

Recommended decomposition:

For each task, report:

clean_baseline_loss
perturbation_induced_loss
clean_sample_noise
perturbed_sample_noise

Example wording:

The total perturbed-prompt loss can be decomposed into the clean-prompt baseline loss and the additional perturbation-induced loss:

perturbed_loss = clean_baseline_loss + perturbation_induced_loss

where:

clean_baseline_loss = 1 - clean_performance
perturbation_induced_loss = perturbed_loss - clean_baseline_loss

This directly answers:

- model/task baseline error: clean_baseline_loss
- perturbation extra effect: perturbation_induced_loss
- repeated-generation instability: sample_noise

Even better case-level decomposition:

For each case:

clean_pass_rate = mean(clean scores)
perturbed_pass_rate = mean(perturbed scores)
case_baseline_loss = 1 - clean_pass_rate
case_perturbation_loss = clean_pass_rate - perturbed_pass_rate

Then classify cases:

1. stable clean correct:

clean_pass_rate = 1.0

The model can solve the clean prompt reliably.
If perturbed_pass_rate drops, the loss is more plausibly perturbation-induced.

2. stable clean wrong:

clean_pass_rate = 0.0

The model already fails the clean prompt.
Perturbation cannot be blamed for this baseline failure.

3. unstable clean:

0.0 < clean_pass_rate < 1.0

The model is intrinsically unstable on this case.
Perturbation effects should be interpreted cautiously and compared against sample noise.

4. perturbation harmful:

perturbed_pass_rate < clean_pass_rate

5. perturbation neutral:

perturbed_pass_rate = clean_pass_rate

6. perturbation helpful:

perturbed_pass_rate > clean_pass_rate

This classification is very useful because aggregate corrected PDR can hide the mechanism.

Recommended metrics:

At task level, report:

- mean clean_baseline_loss
- mean perturbation_induced_loss
- harmful_case_share = share of cases where perturbed_pass_rate < clean_pass_rate
- helpful_case_share = share of cases where perturbed_pass_rate > clean_pass_rate
- neutral_case_share = share of cases where perturbed_pass_rate = clean_pass_rate
- mean clean_sample_noise
- mean perturbed_sample_noise

Recommended interpretation logic:

If clean_baseline_loss is large but perturbation_induced_loss is small:

The task is hard for the model, but perturbations do not add much extra harm.

If clean_baseline_loss is small but perturbation_induced_loss is large:

The model can solve the clean task, but perturbations cause real robustness loss.

If sample_noise is large:

Single-sample PDR is unreliable; repeated sampling is necessary.

If perturbation_induced_loss is smaller than sample_noise:

The perturbation effect is weak relative to ordinary repeated-generation variability.

If perturbation_induced_loss is larger than sample_noise:

The perturbation effect is more likely to be a real robustness signal.

Recommended final framing:

不要把所有 perturbed loss 都归因于 perturbation。

更好的写法是：

模型在 clean prompt 下已有 baseline loss。PDR 的关键不是 perturbed prompt 的总 loss，而是 perturbed loss 相对于 clean baseline loss 的增量。我们将 clean baseline loss 视为模型/任务本身带来的错误，将 corrected PDR 视为 perturbation 额外带来的错误，并用 correctness sample noise 衡量 repeated-generation variability 对该估计的影响。

In English-style formula:

total_perturbed_loss = model_clean_baseline_loss + perturbation_induced_loss

where:

model_clean_baseline_loss = 1 - mean(clean repeated correctness)
perturbation_induced_loss = mean(clean repeated correctness) - mean(perturbed repeated correctness)

sample_noise is reported separately as uncertainty/instability, not as loss.

Meaning of positive values in rq1_pdr_recomputed_v2_corrected_pdr_by_task_bars

User question:

In `rq1_pdr_recomputed_v2_corrected_pdr_by_task_bars`, what does a positive value mean? Does it mean loss appears after perturbation?

Answer:

Yes. A positive corrected PDR value means the perturbation condition has lower repeated performance than the clean condition.

Formula:

corrected_PDR = perturbed_loss - clean_loss

Because:

clean_loss = 1 - clean_performance
perturbed_loss = 1 - perturbed_performance

Therefore:

corrected_PDR = (1 - perturbed_performance) - (1 - clean_performance)
              = clean_performance - perturbed_performance

So:

If corrected_PDR > 0:

clean_performance > perturbed_performance

This means performance dropped after perturbation.
In plain language: perturbation introduced additional loss.

If corrected_PDR = 0:

clean_performance = perturbed_performance

This means no average performance change after perturbation.

If corrected_PDR < 0:

clean_performance < perturbed_performance

This means the perturbed prompt performed better than the clean prompt in this run.
It should not be interpreted as loss. It is a negative loss / improvement.

Example from the figure:

Math reasoning / reordering:

corrected_PDR = +2.40 percentage points

This means:

clean repeated performance - perturbed repeated performance = 0.0240

So after reordering perturbation, math reasoning performance dropped by 2.40 percentage points relative to clean prompt.

Code generation / reordering:

corrected_PDR = -0.80 percentage points

This means:

perturbed performance was 0.80 percentage points higher than clean performance.
It is not a perturbation-induced loss; it is a small improvement in this run.

Important wording:

For positive bars, it is okay to say:

"The perturbation induced additional loss."
"Performance dropped after perturbation."

For negative bars, do not say:

"negative loss appeared"

Better wording:

"The perturbed prompt slightly improved performance relative to the clean prompt."
"The corrected PDR is negative, indicating no harmful perturbation effect in this condition."

For near-zero bars:

Use:

"No measurable average perturbation-induced loss."

Recommended figure caption wording:

Positive corrected PDR values indicate perturbation-induced performance loss relative to the clean prompt. Negative values indicate that perturbed prompts performed slightly better than clean prompts in repeated sampling. Values near zero indicate no measurable average perturbation effect.

Example 1: paraphrase improves math_09

case_id:

promptrobust_pdr_paraphrase_math_09

Perturbation:

paraphrase

Problem:

Four distinct integers a, b, c and d have the property that when added in pairs, the sums 10, 18, 19, 20, 21, and 29 are obtained. What are the four integers in increasing order?

Standard answer:

4, 6, 14, 15

Clean prompt instruction:

Solve the mathematics problem. Put the final answer only on the last line.

Perturbed prompt instruction:

Resolve the mathematics problem. Place the final answer exclusively on the last line.

Clean outputs:

sample 1: 1, 9, 10, 19 -> wrong
sample 2: 3, 7, 8, 21 -> wrong
sample 3: 4, 6, 14, 15 -> correct
sample 4: 3, 5, 15, 16 -> wrong
sample 5: 4, 6, 14, 15 -> correct

Clean pass rate:

2/5 = 0.4

Perturbed outputs:

sample 1: 4, 6, 14, 15 -> correct
sample 2: 4, 6, 14, 15 -> correct
sample 3: 4, 6, 14, 15 -> correct
sample 4: 4, 6, 14, 15 -> correct
sample 5: 4, 6, 14, 15 -> correct

Perturbed pass rate:

5/5 = 1.0

Improvement:

1.0 - 0.4 = +0.6

Explanation:

The clean samples often chose incorrect integer sets whose pairwise sums do not match the required six sums. The paraphrased instruction did not change the math problem, but in this run all five perturbed generations followed the correct reasoning path. This is a clear performance improvement in the sample, but it should be interpreted cautiously as sampling variability plus possible prompt wording effect, not definitive evidence that paraphrasing always helps this problem.
